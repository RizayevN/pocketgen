## Project

PocketGen is an iOS 17+ SwiftUI app (the directory is named `pocketgen`) that pitches *unlimited on-device image generation* — prompts and images never leave the device. Today the actual pixel production is a placeholder; the app is built so a real Core ML diffusion pipeline can drop in without touching the UI.

## Build & project generation

The Xcode project is **generated by [XcodeGen](https://github.com/yonsson/XcodeGen) from `project.yml`** — `project.yml` is the source of truth, not `PocketGen.xcodeproj`. After adding/removing/moving source files or changing build settings, regenerate:

```sh
xcodegen generate
```

Source files are added by directory globbing (`sources: - path: PocketGen`), so new files under `PocketGen/` are picked up automatically on regeneration — no manual project edits.

Build (Simulator, no signing):

```sh
xcodebuild -project PocketGen.xcodeproj -scheme PocketGen \
  -destination 'platform=iOS Simulator,name=iPhone 17,OS=26.2' \
  -configuration Debug build CODE_SIGNING_ALLOWED=NO
```

There is currently **no test target** and no tests. `Info.plist` is checked in (`GENERATE_INFOPLIST_FILE: NO`).

## Architecture

The app is a three-tab shell (Create / Gallery / Settings) over one generation engine behind a protocol. `PocketGenApp` hosts `RootView` and forces `.dark` color scheme app-wide.

- **`RootView` (`Views/`) is the app shell** — a `TabView` of `GenerateView` (Create), `GalleryView` (Gallery), and `SettingsView` (Settings). It owns the single `@StateObject GalleryStore` and injects it into all three tabs, so an image generated on Create immediately appears under Gallery and is counted/clearable from Settings.
- **`GenerationEngine` protocol (`Engine/GenerationEngine.swift`) is the key seam.** It is the only contract between the UI and pixel production: `generate(_:progress:) async throws -> (UIImage, UInt32)`, plus `isReady`. A real on-device pipeline (Apple's `ml-stable-diffusion` Core ML package) only needs to conform to this protocol — no view-model or view changes required. Errors surface as `GenerationError`; `.cancelled` is treated as a non-error.
- **`MockGenerationEngine` is the current implementation** — it fakes step-by-step diffusion progress (`secondsPerStep`) then renders a *deterministic* prompt+seed-derived abstract gradient image with a "MOCK" watermark. Determinism mirrors real seeded diffusion (same inputs → same image, enabling "generate again"). When wiring a real engine, replace the default in `GenerationViewModel.init(engine:)`.
- **`GenerationViewModel` (`@MainActor`, `ObservableObject`)** owns all generation state via a `Phase` state machine (`idle → generating(progress) → finished | failed`). `GenerateView` switches its entire UI on `phase`. The generation `Task` is cancellable via `cancel()`. `engine` is injected (defaults to `MockGenerationEngine()`), which is how previews/tests swap in a fast or fake engine. On success it appends the result to the injected `GalleryStore`.
- **`GalleryStore` (`@MainActor`, `ObservableObject`)** is the generation history — `add(_:)` inserts newest-first, `remove(_:)` drops one, `clear()` empties it (used by Settings). The published array is the synchronous source of truth for the UI; every mutation is mirrored to disk via **`GalleryPersistence`** (`Support/`), an actor that stores one PNG per image plus a JSON metadata manifest under `Application Support/Gallery`, so history survives relaunch (restored asynchronously on launch, appended behind any images generated before the load finishes). Persistence failures are silent — the image stays in the session gallery, it just won't survive relaunch. Previews/tests can point the actor at a different directory.
- **`GenerationRequest` (`Models/`) is the unit the engine consumes** — prompt, negative prompt, `ImageSize` (384/512/768 sq px, matching on-device SD configs), steps (10–40), optional `seed` (nil = random at generate time). Designed to be persistable/replayable later. **`GeneratedImage` (`Models/`)** pairs the rendered `UIImage` with the `GenerationRequest`, resolved `seed`, and `createdAt`, so the UI can show provenance and offer "generate again with same settings".
- **`DeviceCapability` / `DeviceTier`** maps the raw device model identifier (e.g. `iPhone13,2`) to a coarse tier, exposed two ways: an up-front hardware-expectation `banner` on Create, and a short `name` ("Recommended" / "Limited" / "Below recommended") shown in Settings. Unknown identifiers and the **Simulator resolve to `.recommended`** (err toward "you're fine"), so tier-banner logic can't be exercised on the Simulator.

Tabs and supporting pieces (all in `Views/`): `GenerateView` (composer), `GeneratingView` (progress), `ResultView` (save/share), `GalleryView` + `GalleryDetailView` (grid + detail), `ImageActions` (shared save/share buttons), and `SettingsView` (hardware tier, image count + Clear Gallery, on-device privacy note, app version from the bundle). `Support/` holds `PhotoSaver`, `ShareSheet`, and `GalleryPersistence`.
